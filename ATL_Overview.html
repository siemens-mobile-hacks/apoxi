ATL Overview

1 Introduction

APOXI module "ATL (Application Template Layer)" provides basic application management and customization functionality on top of APOXI.

The most important aims of the ATL module are the following:

  • Support factory customization of central parts of the user experience of a phone software
  • Support code size savings for applications

2 ATL Architecture

2.1 ATL Application and Application Plugin

The ATL application is the mandatory base class for an application intending to use ATL concepts. The ATL application plugin is the base class for implementing lightweight add-ons to an ATL application.

2.2 ATL GUI Controls

All ATL GUI controls extend existing APOXI GUI classes. The APOXI window extensions provided by ATL are:

  • ATL Window as mandatory base class for any application window.
  • List Window for displaying data of an APOXI Provider.
  • Menu Window, which is used by the ATL framework for displaying and handling full screen as well as popup menus. There is no need for an application developer to use this class directly.
  • Property Editor, which is the base class for any window to be used for viewing and changing setting values according to the Settings Framework specification.

3 Feature Handling

Features are functional entities provided by ATL applications or plugins. With the ATL Feature Broker, particular features of an application may be exported in order to be available globally and used as menu entries, shortcuts or other event triggered actions.

A feature is identified by a Feature ID, and it is linked to an application which handles the feature. Such an application is called Feature-providing application. When the feature is executed, the feature-providing application executes the functionality corresponding to the feature.

The different types of features provided by ATL are:

  • Context independent Features: Context independent features can be triggered by a menu or any other asynchronous event without the need to supply additional information. These features are typically used to open particular dialogs of applications (e.g. write an SMS, Open Phonebook, ….).
  • Context aware Features: These are features require a feature invocation argument. This argument is the context information, when they are invoked. E.g, to establish a call, a phone number has to be supplied. Then, a context aware feature "Setup Call" is initialized with a phone number as context information, before it is executed.
  • Platform Features: Platform Features are features not linked to a particular application, because they are provided by the platform. As an example, features using a service interface do not need to know which application implements the service.

3.1 Feature Broker

The Feature Broker provides access to all globally available features of a phone. It serves the following purposes:

  • Feature registration: In order to be able to access all features on the phone, feature-providing applications must register their features to the Feature Broker.
  • Feature retrieval: In order for client objects to use a feature, they must request the feature from the Feature Broker by specifying the Feature ID.

3.2 Menu Configuration

ATL menu structures are specified by menu descriptions. A menu description maps application features to menu items of the phone. According to the menu description, ATL creates feature items and item containers, which provide menu display as well as menu hierarchy information. GUI components use them for drawing as well as traversing through hierarchies.

3.3 Menu Window

ATL provides a menu window, which uses a menu description to display a menu node and let the user traverse through the hierarchy. The menu description has to be set by the client. Unavailable menu items are not displayed at all, disabled menu items are displayed as grayed.
The menu window listens for two virtual keys, namely:

  • Select Key triggers the selected menu item.
  • Back Key steps the hierarchy one level up. On level = 0 it closes the window.

4 Context Handling

A context specifies a particular state of the user interface. For each context, project configuration is able to assign a specific set of user experience elements.
A context is uniquely identified by a Context ID. In ATL, at least each window within the system possess a context. It is also possible to define more than one context for a window according to particular states of the window. The configurable user experience elements are key mappings, Window properties like window title, softkey assignments for the softkey bar.

4.1 Key Map Configuration

With context handling, the set of virtual keys, which are not represented by existing hardware keys, can be emulated per every context. The available key mappings are:

  • Mapping of hardware keys to virtual keys: The hardware keys can be mapped to virtual keys for doing different actions. For example, a particular hardware key may act as the virtual key Delete Key in one context and as virtual key Cancel Key in another context.
  • Mapping of virtual keys to trigger a feature: A virtual key can be mapped to trigger a particular feature. This virtual key is called feature key. With the Feature ID specified for the feature key, the key handling is able to fetch the feature from the Feature Broker and execute it if it is available. It is also possible to map a long or very long, or a sequence of hardware keys to a feature.

4.2 Softkey Configuration

The softkey configurations are described in the following:

  • Feature assigned to Softkey: If the Context Manager detects a feature configuration for a softkey, it fetches the feature from the Feature Broker and if that feature is available applies it to the global softkey bar. When the softkey is pressed, the mapped feature is executed.
  • Context Menu assigned to Softkey: Menus are identified by Menu ID. The Context Manager can use Menu IDs in order to fetch the corresponding menu description from the Feature Broker. A special feature indicating the text and possibly icon of the menu root node must be used to be applied to the global softkey bar. If that feature is triggered, a menu window parameterized with the menu description of the context menu will be opened.

4.3 Context Switching

A window based on the ATL window base class is contextable. If an ATL window becomes focused, the context of the window is set to the Context Manager. The context manager is responsible to fetch the corresponding context configuration data from the project configuration and to apply the configured user experience elements to the active context.

5 Settings Framework

The ATL Settings Framework provides properties. A property represents a particular setting maintained by an ATL application. Properties are special features and are available globally. They can be used as menu items, whereby a matching property editor must be configured. If the property item is triggered, the configured editor is used to view and change the property value. Storage of a changed property value is handled by the ATL application responsible for the property.
