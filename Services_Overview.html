Services Overview

1 Introduction

A service in APOXI is used by applications to provide functionality to other applications. This avoids dependencies between applications. In software engineering terms this is called loose coupling. Services initiate actions in other applications.

2 Service Broker

Every service has a Global Unique ID. So there must be one instance which oversees all available services within APOXI. This instance is the Service Broker. It knows about all services, and applications can query it for specific services. There are some analogies in this concept, when comparing it to other well-known object management models such as CORBA or Microsoft's COM.

Each service needs to register to the service Broker inorder to be available for other applications. The calling application has to make sure that the requested service is available before executing a service-call. However an available service can be temporarly inaccesible to the user.

3 Service Concept

The following example illustrates the service concept:

  1. A WAP Browser shows a page with a mail to link
  2. The user clicks on it
  3. The WAP application asks the Service Broker whether there is a SendMail service available
  4. If available, invoking the service results in opening a "Send Mail" Window


The WAP application in this example does not know which object has implemented the SendMail service. The Service Broker hides this information. The WAP application just knows about the interface the service is offering. To be independent from the service mplementation, the WAP application should not have knowledge about the specific service-implementing class (object) at compile-time.

Services are mainly used to exchange data or invoke actions at a foreign application without the need of returning a result.

Asynchronous execution is realized by internally forwarding the service-call by a post-message to the service implementing the application itself.

It is recommended to implement the service interface utilizing the facade-design pattern. The facade pattern provides a simple unified interface and the lower level implementation is Application depended and hidden to the user. E.g. the SMS and EMS service implementations use the same common service interface.

4 Implementing a Service

All the MMI services available in the system are defined in the file MmiServices. Each service has a system wide unique ID. All the service IDs are defined in the MmiServices file.
